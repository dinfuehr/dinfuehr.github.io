Stack Unwinding in JavaScriptCore

November 15, 2018 6-minute read

JavaScriptCore (JSC, WebKit’s JavaScript engine) needs to be able to unwind the call stack e.g. printing stack traces or exception handling. This is not so simple since JS and native/C++ can be arbitrarily intertwined and JS call frames do not match the system’s calling convention. While JSC fully controls JS stack frames and knows how to unwind them, this isn’t the case with C++ stack frames. Such stack frames are set up by the C++ compiler and are dependent on the flags/system. This post describes how JSC manages to unwind the stack in the presence of native stack frames.

JSC’s Calling Convention Link to heading

JSC uses its own calling-convention for JS-Functions. This convention is used by the interpreter, the baseline JIT and its optimizing compilers DFG and FTL, such that JS-functions emitted by different compilers are able to interoperate with each other.

All arguments are passed on the stack, there are also some additional values passed as argument.

Calling JS-Functions from C++ Link to heading

Since JSC’s calling convention is quite different to the C ABI, C++ can’t directly call JS-Functions compiled by JSC. JSC needs a short trampoline to translate between them: vmEntryToJavaScript.

C++ doesn’t call the JS function directly - it calls vmEntryToJavaScript instead. This trampoline has quite some responsibilities:

  • allocates and initializes a VMEntryRecord on the stack, that is later used for stack unwinding,
  • it pushes the arguments on the stack but also everything else that is required by JSC’s calling convention,
  • executes the actual JS-Function,
  • and after function execution it removes the stack frame and returns to the caller.

But note that unlike e.g. the JVM, JSC doesn’t save and restore callee-saved registers here. JSC makes use of callee-saved registers in its internal calling convention unlike the JVM. (JSC only needs to do this for ARMv7 where callee-saved registers in the C ABI are used as caller-saved registers in JSC’s calling convention).

Calling C++ from JS Link to heading

JS code can call back to C++ by invoking JS functions which are actually implemented in C++ under the hood. Even though these functions are written in C++, they still need to adhere to JSC’s calling convention. Here is an example for a JS-function implemented in C++:

JSC_DEFINE_HOST_FUNCTION(functionSleepSeconds, (JSGlobalObject* globalObject, CallFrame* callFrame))
{
    VM& vm = globalObject->vm();
    auto scope = DECLARE_THROW_SCOPE(vm);

    if (callFrame->argumentCount() >= 1) {
        Seconds seconds = Seconds(callFrame->argument(0).toNumber(globalObject));
        RETURN_IF_EXCEPTION(scope, encodedJSValue());
        sleep(seconds);
    }
    
    return JSValue::encode(jsUndefined());
}

Note that this functions gets exactly two arguments: the global object and the call frame. The CallFrame* is a pointer into JSC’s stack frame. Through this pointer C++ can access all the value in the JS stack frame: So e.g. argument count, the receiver and all arguments (for example callFrame->argument(0)).

Calling native methods like this requires another trampoline: nativeCallTrampoline

This trampoline first sets up the two-arguments passed to the native method. It also stores the current call frame in VM::topCallFrame (this will be discussed later in more detail). It then invokes/calls that native function. When that function returns, it checks VM::m_exception whether the C++-function has thrown an exception and if so calls the exception handler. Afterwards it returns to the caller JS function.

Stack Unwinding Link to heading

JS and C++-stack frames can be arbitrarily intertwined, JSC therefore needs a way to safely unwind the stack. JSC doesn’t have any knowledge about C++ stack frames, so it skips that part of the stack at once (indepedent of how many C++ stack frames there are). For JS functions it is possible - and required - to unwind function by function.

JSC stores the begin and end of the last active region of JS stack frames: VM::topCallFrame and VM::topEntryFrame. Unwinding that region of the stack just follows the saved frame pointer on the stack until JSC reaches topEntryFrame. As soon as we have reached that entry frame, JSC knows the next frame is a C++ frame.

The problem now is that JSC doesn’t know how large that stack frame is or where its caller frame pointer is stored. We already discussed that calling from native code to JS sets up a VMEntryRecord on the stack. That record stores (among other stuff) m_prevTopCallFrame and m_prevTopEntryFrame. This allows JSC to skip all C++-functions at once by continuing unwinding from m_prevTopCallFrame. Stack unwinding can now continue from that position until it m_prevTopEntryFrame. At that point JSC can check the next VMEntryRecord for m_prevTopCallFrame and m_prevTopEntryFrame. This process is repeated until m_prevTopCallFrame is null.

When calling into JS from C++, JSC therefore stores the current VM::topCallFrame and VM::topEntryFrame in the VMEntryRecord as m_prevTopCallFrame and m_prevTopEntryFrame. VM::topCallFrame and VM::topEntryFrame are initialized to the current call frame (of this intermediate function) and the actual callee’s call frame. vmEntryToJavaScript/vmEntryToNative also need to restore the previous values when returning from the JS-Function.

Short recap: The JS parts are unwound frame by frame, while the C++-parts are skipped all at once. Unwinding starts at VM::topCallFrame.

Unwinding also needs to support inlined frames: DFG and FTL can inline a JS-function into another, however on the stack this is now one combined stack frame. Nevertheless unwinding should still be able to recover inlined functions. Otherwise stack traces would be missing some function calls.

Callee-saved registers Link to heading

As already mentioned JSC uses caller- and callee-saved registers in its calling convention. When unwinding the stack JSC might also need to inspect values in callee-saved registers, JSC therefore needs to be able to check where these registers where saved. codeBlock->calleeSaveRegisters() returns a list with the offsets of the stored registers on the stack.

LLInt/baseline and DFG use predefined callee-save sets. FTL has the most flexibility and can use a larger set of ABI callee-saves depending on generated code.

When unwinding needs callee-saved register contents, JSC snapshots them into VMEntryRecord via copyCalleeSavesToVMEntryFrameCalleeSavesBuffer and later restores them via restoreCalleeSavesFromVMEntryFrameCalleeSavesBuffer. This is used for paths like exception handling and OSR, not for the regular JS->native call.

Saving all possibly used callee-saved-registers in the current VMEntryRecord seems to be needed for throwing exceptions and On-Stack-Replacement (OSR). The other case where we would definitely need this is for determining GC roots - but since JSC uses a conservative GC the stack isn’t actually traversed but just examined word by word for references into the heap.

The main takeaway probably is that all callee-saved registers are only saved for certain actions (throwing exceptions and OSR), but not on each call from JS to native code. This makes it faster to call native functions. VMs with precise GC need to store callee-saved registers each time they call into native code.

On ARMv7 some callee-saved registers are saved and restored when calling from C++ into JS by vmEntryToJavaScript and vmEntryToNative: This is because JSC internally uses them as caller-saved registers instead. Having these thunks at the JS to C++-boundaries lets JSC use registers differently than originally intended by the system’s ABI.

Conclusion Link to heading

Much of the information above is quite implementation-specific, nevertheless it is still very interesting how JSC manages to traverse the stack since quite a lot of functionality is affected by this (e.g. dumping stack traces, exception handling, etc.).
